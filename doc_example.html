Tag data processing

May 26, 2022

The only information that each tag conveys is its EPC, which is just 96 bits. Many of these are used by WISP for things like the WISP ID, the hardware revision number and the type of WISP tag. There are 64 remaining bits that WISP uses to convey real data, be it accelerometer data, pixels of a photo or temperature readings. These need to be parsed to be useful for analysis and each tag needs to be parsed differently. Finally, registering new tag types (and their associated parsers) needs to be fairly easy for a user.

Performance improvements with throttling

May 18, 2022

As far as I can tell, it’s hard to prevent slow widget rendering from having an effect on the rate tags are processed. The best I can do right now, is to make sure widgets are re-rendered at a reasonable rate, rather than on each new tag.

I’ve implemented simple throttling for widget updates. Widgets won’t update more than 4 times a second, which is more than fast enough for the kinds of visualization applications I have been considering. Of course, tags are still received and processed as fast as sllurp will allow, which appears to be around 100 tags/second.

Now, the UI seems to be unbothered by a high inventory rate (up to 100 tags/sec) or tag buffer size (tested up to 20k tags).

Ideally this widget update rate could be controlled dynamically by the UI. This would allow it to operate as fast as possible when it can, rather than be fixed at 4 updates/second.

Automatic pausing for slow data

May 13, 2022

While I work on optimizing the app to allow for faster tag reads (and individual widgets from contributing to slowing down the entire UI), I wanted to add a feature to measure the amount of time that the UI is running behind for each tag. Each time a tag is recorded, it computes the delay between the time the Python recorded seeing the tag to when the tag was recorded by the UI.

If it’s higher than a threshold, 250 ms for the time being, the new feature temporarily pauses new tags from being accepted for a set time, 1 second right now. It also alerts the user that the UI had to pause and that some tags may have been lost.

If it happens that the UI simply can’t update fast enough, this pausing will just continue happening repeatedly, but it signals to the user that something is wrong and data is being lost while also saving the UI from crashing completely in a worst case scenario.

Right now, this only happens when either there are a large number of tags in the cache (~100k) or if resource intensive renders are happening too often (for example putting ~10 graphing utilities on the dashboard). The latter is a problem that I’m hoping to address next. I need to find a way to separate tag handling from widget rendering so a slow widget doesn’t end up slowing everything down.

Tag filtering

May 5, 2022

In the app’s current form, widgets receive data from all the tags that the reader sees. Unfortunately, especially in a lab setting, there are often stray tags that could make analyzing the data of interest harder. Thankfully, WISP tags use the last 4 bytes to encode a WISP ID. We can use this to filter just the tag(s) of interest.

To avoid unnecessarily sending tags from Python to the JS frontend, I implemented the filters directly in the Python. Each time the whitelist and blacklist filters are changed on the front-end, they are sent to Python which begins applying the new filter to incoming tags. The filters can be modified any time, even during an inventory to make capturing real time data easier. This approach does have the drawback that tags that are filtered out are never collected, so they can’t be restored.

The filters are applied in order, so if there is a whitelist, only the whitelist is used to filter tags. If not, the blacklist is used. Here’s what the UI for filtering looks like:

WISP app development

Apr 11, 2022

Since the app receives so much data each second from the Python backend, we must be careful to handle this data as fast as we can and limit the amount of the UI that has to be rerendered each time we receive a tag. All this tag data also has to be close to globally accessible (each widget and all the connection management stuff needs it). React’s Context is pretty convenient for this kind of data.

A Provider component sits as a parent of most of the app which collects this data and stores it in its state. It provides access to its state to any children that are Consumers to the Context. This means that only the content that requires an update gets one when a new tag is recorded.

As I began working on some widgets, I realized that many different hypothetical widgets would end up having to do a lot of repeated calculations on the raw tag data. For example, a table of the most recent tags, would not need access to every single tag recorded and it could safely group tags with the same WISP ID. For this reason, widgets have the option to either subscribe to an array of every single tag recorded (for example, a graphing widget would use this kind of data), or just the most recent versions of tags with unique WISP IDs.

I2C library

Apr 1, 2022

The I2C library provided by TI for the MSP430 is large and looks cumbersome to use. I found this article when I was looking for alternatives. This library operates with just two functions,  init  which sets the speed of the bus and  send_sequence  which can send any sequence of bytes and read/restart operations to the bus. It uses the Bus Pirate convention for the sequence which, conveniently, closely matches the actual addresses being sent on the bus, making it easier to debug.

For example, this sequence will do the following:

 {0x38, 0x0c, I2C_RESTART, 0x39, I2C_READ} 

  1. Issue start (included automatically)
  2. Write  0x0C  to device  0x1C 
  3. Do not release the bus and issue a repeated start
  4. Read one byte from the device  0x1C
  5. Issue stop (included automatically)

Importantly, the  0x38  and  0x39  come from the device address,  0x1C  shifted with either a 0 for writing, or a 1 for reading.

For the ICM20602 with an address of  0x68 , we would write with  0xD0  and read with  0xD1  ( (0x68<<1)|1 ).

The library (if I’m able to get it running on the hardware) seems like a pretty good solution, so I’ll see how it runs next time I have access to a scope or logic analyzer for debugging.

As an aside, I have been using one of the scopes at the UW MILL with the I2C decoder feature mostly for logic debugging. However, I think I’m going to try this, almost comically inexpensive ($13 shipped) USB logic analyzer when it arrives. According to the listed specs and reviews, it should be way more than enough for the simple I2C stuff I’ve been working on and it should work with industry standard software. We’ll just have to see though.

Desktop app backend

Feb 20, 2022

The new WISP app will rely on the Sllurp library to handle communication between the reader and a front-end interface. This means that a Python backend will be required.

Today, Electron is a popular way to build cross-platform applications. Electron apps traditionally use a built-in version of the Chromium web browser to render the front end interface. Going this route would have a few drawbacks:

  • The total package size becomes quite large due to the included browser
  • The front-end cannot communicate directly with the python Sllurp API

To solve both of these, the Python EEL library seems like a good option. It replaces the Node JS backend of Electron with Python. This backend can directly use the Sllurp library to talk with an RFID reader, which will simplify the app as a whole.

Secondly, EEL avoids the large stack size of an included Chromium browser by using the default system web-view. For most modern operating systems, this should be more than sufficient for our relatively simple user interface.

Setting up the Impinj reader

Jan 31, 2022

I picked up an Impinj reader, WISP 5 and debugger from the lab today and began setting it up in my dorm room.

Due to my limited access to the university-managed internet, connecting the Impinj directly to my computer was really my only option for getting data from it. Here’s what I learned:

  • Documentation for this Impinj model is very limited online due to it’s mainly industrial uses and the fact that it’s no longer produced.
  • On Windows (and not Ubuntu for some reason), an ethernet crossover connector (rather than a standard ethernet cable) is required between the computer and reader. A crossover connector reverses the pin order of RJ-45 connector on one side of the cable. I’m not exactly sure why this is the case, but I did find it in the little documentation I could find. I do not know why this isn’t the case on Ubuntu however.
  • You must explicitly set the computer’s IP address on this local network to be on the same subnet of the RFID reader, otherwise, while the computer will be able to identify the IP address of the hostname (speedway-XX-XX-XX.local, where the XXs are the last 6 digits of the MAC address)
